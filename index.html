The project

The aim of this project is to make some experiment with graph embedding and link prediction starting from a raw dataset not yet semantically meaningful. It was developed as a final examination for the Knowledge Representation and Extraction course of the Digital Humanities and Digital Knowledge master Degree at the University of Bologna held by professor Aldo Gangemi and Andrea Nuzzolese. The goal is to show that it is possible and fairly easy to transform non-linked data in a linked form, and that allows for many interesting options. DBpedia and Wikidata already offer a wide amount of entities related to music, but their properties are often very limited for all the musical records that are not much popular. Using a much more wide dataset, which is more specialized in musicla records, allow to make more interesting and accurate predictions. In the field of musical recors, this could lead to a system which not only automatically predic the genres, descriptors and features of an album records, but also the probability of its likeness, taking into account the average scores given by the user.

In order to accomplish such task, the first step necessary was to gather a raw dataset that could be formalized in a set of triples. I decided to work with a musical dataset regarding records and artists, so I extracted raw data from the RateYourMusic.com Website. RateYourMusic is a large musical database in which users are able to autonomusly enter musical records and to label them according to various properties, such as its artist, the its publication date, the musical genres, a range of descriptors and so on. For the purposes of this project, about 22000 musical records were extracted, ranging from the most popular to the lesser ones.

Then a simple and small ontology was developed using Protegé as a software. The ontology links an album record with its main properties, namely the artist that made the record, its main genre, its secondary genre, some descriptors and so on. Moreover, since one of the goals was to allign the dataset with Wikidata, the artist subclasses was subdivded in two different classes, i.e. humans (songwriters, performers, musicians, etc.) and musical groups. Those subclasses have properties as well, such as a birthplace for humans, a list of members for bands and so on. The list of properties for artists was completely arbitrary and it can be extended any time. You can download the Protegé file here.

From a coding point of view, Python was used in order to exploit the vast amount of libreries avaiable for handle data. Pandas was used to giving a first look at the data, cleaning a bit, eliminating duplicates and getting rid of some unecessary properties. It was also important to allign the entries with the already existing entries of a linked open datataset, such as DBPedia or Wikidata. Thus, it was developed a set of functions that are able to automatically extract information about a specific entetity (using a SPARQL query) and put them in the main dataset. Once the dataset was alligned (in this case with Wikidata, it became possible to serialize in subject-predicate-object triples. Using an RDF library, it became possible to create a set of triples that linked every musical entity with its properties. Here you can see an example extracted from the graph using the Turtle syntax:

With the Turtle file, it became possible to make some experiment with graph embedding. For this task, two Python libreries were individuated:Pykeen and RDF2Vec. However, after an initial phase of testing, RDF2Vec was discarded due to the the difficulties related to set a proper work enviroment. Moreover, Pykeen offers a built in function for link prediction, thus being much easier to set up for whom do not have prior knowledge in machine learning. For the training phase the dataset was automatically splitted in two (with a 80/20 proportion) and fed to the pipeline function without changing any of the default settings (except with the epoche numbers, which was set at 1500). You can download the results of the traning here.

All the data, including code and the trained model, can be found at the GitHub repository.

The steps

The complete Python code can be explored at the Colab repository.

  • Gather raw data

    The data were extracted from the charts using a JavaScript application developed by Simone del Deo which converts them into a raw .csv file.

  • Clean the data and link them to Wikidata

    In order to create a dataset semantically meaningful, some basic cleaning was requried. Pandas was used to eliminate duplicates, excluding useless columns and linking each entry with the realted Wikidata entry.

  • Create a Turtle file

    The final .csv table contained more then 22K entries, which were then linked as a RDF file using the Turtle syntax. The final Turtle file contained more than 200K relations, linking records with their descriptor and the related artists with (some of) their Wikidata properties.

  • Graph Embedding

    Once the linked dataset was built, it became possible to do some interesting operations on it. Pykeen was used as a library to create a graph embedding of the dataset. In order to exploit the Pykeen functionalities, the graph was converted into a tabular file, each row representing a triple. This allowed to perform many significant operations on the data, such as link predicition and sentimental analysis.

Conclusions and further developments

The aim of the present project was to extract raw data and transform them in proper knowledge. Thanks to graph embedding it is possible to exploit this knowledge to gather new knowledge through machine learning algorithms tecniques. The current dataset seems to behaeve well for most of the link predictions tasks. Indeed, there are clear differences between the different kind of predictions: for istance, the artist property was much harder to predict than the others. However, for what it concerns the genres and descriptors, the system performs actually pretty well and it is able to make significul predictions. It is also interesting to notice that even when the guessed were wrongs, usually there were pretty close to the actual guess. The system predicted artists that share something with the real record's artist, being that for origins, influences or musical genres. For what it concerns the properties inherited from Wikidata, the system seems pretty reliable as well: it always able to distinguish between the kind of artist (i.e. whether it is a single human being or a band) and to predict some of their properties.

There is no doubt that the current system could be enhance on many ways. First of all, a selection among the whole bunch of properties of the Wikidata dataset was made. Exploing all the individual properties could lead to improvemtns in predicting certain properties; moreover, it could be possible to link the band members with their Wikidata entry and gather further knowledge about the main entity. Secondary, here the average rating given by the users was not implemented because it would little sense to predict an integer using grph embedding. However, having at hands an average rating for all this album records could lead us to implement a seprate algorithm to automatically predict a score (which of course should be ponderate with the number of ratings). This could help in creating an accurate software for suggesting records to final users.
